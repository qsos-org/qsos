Le système QSOS fournit des endpoints REST pour accéder programmatiquement aux données d'évaluation.

## Authentification

Les requêtes nécessitant une authentification utilisent OAuth par session via GitHub ou GitLab.

**Flux d'authentification :**

1. Rediriger vers `/auth/github` ou `/auth/gitlab`
2. S'authentifier avec le fournisseur
3. La session utilisateur est automatiquement établie
4. En cas de succès, l'utilisateur est redirigé vers `/profile`

**Accéder à la session courante :**

La session utilisateur courante est accessible côté client via le composable `useUserSession()` fourni par `@nuxtjs/auth-utils`. Cela permet d'accéder à :

- `user.name` - Nom d'affichage de l'utilisateur
- `user.email` - Adresse email de l'utilisateur
- `user.roles` - Tableau des rôles (ex: `["admin"]` si admin, sinon `[]`)
- `loggedInAt` - Timestamp de la connexion

> **Note :** `roles` contient `["admin"]` si l'email de l'utilisateur est listé dans le fichier `roles.json` du serveur, sinon un tableau vide `[]`.

## Gestion des erreurs

L'API utilise les codes de statut HTTP standard :

| Code | Signification | Exemple |
|------|---------|---------|
| `200` | Succès | Requête GET / PUT |
| `201` | Créé | Requête POST réussie |
| `204` | Pas de contenu | DELETE réussi |
| `400` | Mauvaise requête | Données invalides |
| `401` | Non autorisé | Session manquante ou expirée |
| `403` | Interdit | Pas propriétaire ou admin |
| `404` | Non trouvé | Ressource inexistante |
| `409` | Conflit | Version en doublon |
| `500` | Erreur serveur | Erreur serveur |

**Format de réponse d'erreur :**
```json
{
  "statusCode": 400,
  "statusMessage": "Bad Request",
  "message": "Description détaillée de l'erreur"
}
```
