QSOS organizes software into **categories** to provide more relevant evaluation criteria. Each software type has specific evaluation grids tailored to its unique characteristics. This is **Phase 1** of the methodology.

---

## STEP 1: Create a Type

A **software type** groups comparable solutions that address the same need, such as *Web frameworks*, *Databases* or *Monitoring tools*.

::doc-image
---
src: /images/software-types/create-software-type-en.png
alt: Creating a software type
caption: Figure 1 — Software type creation form
maxWidth: 350px
---
::

Fill in a clear **name** and a **description** that defines the scope of the type. You can also add an **icon** for visual identification in lists.

<details>
<summary>View JSON</summary>

```json
{
  "uid": "web_frameworks",
  "name": "Web frameworks",
  "description": "Development frameworks for web applications",
  "icon": "https://example.com/icon.png",
  "creatorEmail": "alice@company.fr",
  "createdAt": "2025-09-08T13:44:00.578Z",
  "updatedAt": "2025-09-08T13:44:00.578Z"
}
```

</details>

---

## STEP 2: Add a Software

Once the type is created, add the software you want to evaluate and compare.

::doc-image
---
src: /images/software-types/create-software-en.png
alt: Creating a software
caption: Figure 2 — Software creation form
maxWidth: 350px
---
::

For each software, fill in its **name**, a short **description**, its **license** in SPDX format (e.g., MIT, Apache-2.0), its official **URL**, and optionally a **demo URL** and an **icon**.

<details>
<summary>View JSON</summary>

```json
{
  "uid": "react",
  "name": "React",
  "description": "Library for web and native user interfaces",
  "licenseId": "MIT",
  "url": "https://react.dev/",
  "demoUrl": "https://react.dev/learn",
  "icon": "icon.png"
}
```

</details>

---

## STEP 3: Add a Version

Each software can have multiple **versions** corresponding to different project states. Evaluations are performed on a specific version.

::doc-image
---
src: /images/software-types/add-software-version-en.png
alt: Adding a version
caption: Figure 3 — Software version form
maxWidth: 350px
---
::

Fill in the **version number** (e.g., `19`, `3.2.1`) and a **summary** of major changes.

<details>
<summary>View JSON</summary>

```json
{
  "version": "19",
  "summary": "Introducing Actions and Server Components",
  "dateAdded": "2025-09-08T13:44:00.578Z"
}
```

</details>

---

## Best Practices

- Clear and descriptive names for types
- Official SPDX licenses only
- One version = one project state (don't mix branches/forks)
- Reuse existing types rather than creating new ones
- Document software icon for visual recognition
