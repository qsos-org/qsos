The QSOS system provides REST API endpoints for programmatic access to evaluation data and functionality.

## Authentication

All API requests requiring authentication use session-based OAuth via GitHub or GitLab.

**Authentication Flow:**

1. Redirect to `/auth/github` or `/auth/gitlab`
2. Authenticate with the provider
3. User session is automatically established
4. On success, the user is redirected to `/profile`

**Getting Current Session:**

The current user session is accessed client-side using the `useUserSession()` composable provided by `@nuxtjs/auth-utils`. This gives access to:

- `user.name` - User's display name
- `user.email` - User's email address  
- `user.roles` - Array of roles (e.g., `["admin"]` if admin, otherwise `[]`)
- `loggedInAt` - Timestamp of login

> **Note:** `roles` contains `["admin"]` if the user's email is listed in the server's `roles.json` file, otherwise an empty array `[]`.

## Software Types

### Get All Software Types
GET /api/software-types

**Response (200 OK):**
```json
[
  {
    "uid": "web_frameworks",
    "name": "Web Frameworks",
    "description": "Development frameworks for web applications",
    "icon": "https://example.com/icon.png",
    "creatorEmail": "alice@company.com",
    "createdAt": "2025-09-08T13:44:00.578Z",
    "updatedAt": "2025-09-08T13:44:00.578Z"
  }
]
```

### Create a software type
POST /api/software-types

→ **Authentication required**

**Request Body:**
```json
{
  "name": "Web Frameworks",
  "description": "Development frameworks for web applications",
  "icon": "https://example.com/icon.png"
}
```

> **Note:** `icon` is optional. `uid`, `creatorEmail`, `createdAt` and `updatedAt` are auto-generated.

**Response (201 Created):**
```json
{
  "uid": "web_frameworks",
  "name": "Web Frameworks",
  "description": "Development frameworks for web applications",
  "icon": "https://example.com/icon.png",
  "creatorEmail": "alice@company.com",
  "createdAt": "2025-09-08T13:44:00.578Z",
  "updatedAt": "2025-09-08T13:44:00.578Z"
}
```

## Error Handling

The API uses standard HTTP status codes:

| Code | Meaning | Example |
|------|---------|---------|
| `200` | Success | GET / PUT request |
| `201` | Created | POST request successful |
| `204` | No Content | DELETE successful |
| `400` | Bad Request | Invalid data |
| `401` | Unauthorized | Missing or expired session |
| `403` | Forbidden | Not owner or admin |
| `404` | Not Found | Resource not found |
| `409` | Conflict | Duplicate version |
| `500` | Internal Server Error | Server error |

**Error Response Format:**
```json
{
  "statusCode": 400,
  "statusMessage": "Bad Request",
  "message": "Detailed error description"
}
```
