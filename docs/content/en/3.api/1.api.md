The QSOS system provides REST endpoints to programmatically access evaluation data.

## Authentication

Requests requiring authentication use OAuth via session with GitHub or GitLab.

**Authentication flow:**

1. Redirect to `/auth/github` or `/auth/gitlab`
2. Authenticate with the provider
3. User session is automatically established
4. On success, user is redirected to `/profile`

**Access current session:**

The current user session is accessible on the client side via the `useUserSession()` composable provided by `@nuxtjs/auth-utils`. This allows access to:

- `user.name` - User display name
- `user.email` - User email address
- `user.roles` - Array of roles (e.g., `["admin"]` if admin, otherwise `[]`)
- `loggedInAt` - Login timestamp

> **Note:** `roles` contains `["admin"]` if the user email is listed in the server's `roles.json` file, otherwise an empty array `[]`.

## Error Handling

The API uses standard HTTP status codes:

| Code | Meaning | Example |
|------|---------|---------|
| `200` | Success | GET / PUT request |
| `201` | Created | Successful POST request |
| `204` | No Content | Successful DELETE |
| `400` | Bad Request | Invalid data |
| `401` | Unauthorized | Missing or expired session |
| `403` | Forbidden | Not owner or admin |
| `404` | Not Found | Resource does not exist |
| `409` | Conflict | Duplicate version |
| `500` | Server Error | Server error |

**Error response format:**
```json
{
  "statusCode": 400,
  "statusMessage": "Bad Request",
  "message": "Detailed error description"
}
```
